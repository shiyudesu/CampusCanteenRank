## 1. main.go 逐行拆解

### 1.1 import 区（main.go:3-18）

这几行决定了入口依赖：

1. config：加载配置。
2. migration：执行 SQL 迁移。
3. logger：初始化日志。
4. 各 repository 包：构建 MySQL/Redis 仓储。
5. router：组装 Gin 引擎。
6. gorm/mysql、redis：持久化与缓存依赖。

你写入口时，import 区就是“系统能力声明”。

### 1.2 main 函数（main.go:20-38）

按顺序看每一行：

1. cfg, err := config.Load()
作用：读取运行配置。如果失败，服务直接退出。

2. logpkg.Init(cfg.LogLevel)
作用：初始化全局日志级别。

3. logpkg.SetSensitiveFields(cfg.LogSensitiveFields)
作用：配置日志脱敏字段。

4. buildRepositories(cfg)
作用：构建 MySQL+Redis 仓储，失败直接返回错误（fail-fast）。

5. defer cleanup()
作用：释放资源（如 Redis 连接）。

6. router.NewEngineWithAllRepositories(...)
作用：把所有仓储注入到路由，构建 gin.Engine。

7. r.Run(":" + cfg.ServerPort)
作用：启动 HTTP 监听。

关键点：main 不写业务逻辑，只做“装配和启动”。

### 1.3 buildRepositories（main.go:45 起）

这是全项目最关键的装配函数之一。

#### 第一步：判定依赖是否齐全

- MYSQL_DSN 和 REDIS_ADDR 缺一不可。
- 任一缺失直接返回错误，不做内存降级。

#### 第二步：初始化 MySQL

- gorm.Open(mysql.Open(mysqlDSN), ...)
- 初始化 sqlDB 句柄用于失败时显式关闭。

#### 第三步：执行迁移

- migration.ApplySQLMigrations(db)
- 失败时关闭连接并返回错误。

这里体现“启动即迁移”策略。

#### 第四步：初始化各领域 MySQL 仓储

按顺序构建：

1. userRepo
2. stallRepository
3. commentRepository
4. mysqlRankingRepository

任一步失败都返回错误并执行资源清理。

#### 第五步：初始化 Redis

1. redis.NewClient
2. Ping 检查连通性
3. NewRedisRefreshTokenRepository
4. NewCachedRankingRepository

最后 rankingRepository 被包装成缓存装饰器。

#### 第六步：返回 cleanup

返回一个函数用于关闭 redisClient。

### 1.4 清理函数与资源生命周期

buildRepositories 返回 cleanup 闭包，至少负责：

1. 关闭 redis client。
2. 关闭 mysql sqlDB。

这保证了启动失败和退出路径都不会泄漏连接。

## 2. config.go 逐行拆解

目标：你能自己写出“环境变量 > 配置文件 > 默认值”的配置系统。

### 2.1 RuntimeConfig 结构

核心字段：

1. ServerPort
2. JWTSecret
3. MySQLDSN
4. RedisAddr / RedisDB / RedisPassword
5. RedisRefreshPrefix / RedisRankingPrefix
6. LogLevel / LogSensitiveFields

这是系统运行参数的唯一入口。

### 2.2 Load 函数关键步骤

1. new viper 实例。
2. 设置默认值（如端口 8080）。
3. 绑定环境变量。
4. 尝试读取配置文件（存在则覆盖默认值）。
5. 解析敏感字段列表。
6. 组装 RuntimeConfig 返回。

重构后新增安全约束：

1. JWT_SECRET 必填。
2. JWT_SECRET 最小长度 32。
3. 不再提供开发默认密钥。

### 2.3 parseSensitiveFields

这个小函数看起来简单，但很实用：

1. 把逗号分隔字符串拆分。
2. 去空格。
3. 去空项。
4. 去重。

这类输入清洗在工程里非常常见。

## 3. migrator.go 逐行拆解

目标：你能自己实现一个简化版 SQL 迁移执行器。

### 3.1 ApplySQLMigrations

每一步含义：

1. 判空 db。
2. resolveMigrationDir：找到 migrations 目录。
3. listUpMigrationFiles：收集所有 .up.sql 并排序。
4. 逐文件读取内容。
5. splitSQLStatements：按分号拆 SQL。
6. db.Exec 逐条执行。

### 3.2 resolveMigrationDir

查找顺序：

1. MIGRATIONS_DIR 环境变量指定目录。
2. server/migrations
3. migrations
4. ../migrations

这是为了兼容不同启动路径。

### 3.3 listUpMigrationFiles

1. 读取目录项。
2. 过滤目录。
3. 只保留 .up.sql。
4. sort.Strings 保证执行顺序稳定。

### 3.4 splitSQLStatements

1. 按 ; 分割。
2. trim 空白。
3. 过滤空语句。

注意：这版分割器适合简单 migration，不适合包含复杂存储过程定义的 SQL。

## 4. router.go 逐行拆解

目标：你能理解依赖注入和路由注册是如何组合的。

### 4.1 三个构造入口

1. NewEngine(secret)
2. NewEngineWithRepositories(secret, userRepo, refreshRepo)
3. NewEngineWithAllRepositories(...)

重构后约束：

1. NewEngine 与 NewEngineWithRepositories 明确 panic，避免误用。
2. 统一使用 NewEngineWithAllRepositories 并显式注入全部仓储。

### 4.2 中间件装配顺序

在 NewEngineWithAllRepositories 中：

1. TraceID
2. RequestLogger
3. Recover

顺序意义：

1. 先给请求打 trace id。
2. 再记录日志。
3. 最后兜底 panic。

### 4.3 仓储注入校验

router 在构造时会检查全部仓储是否注入。

1. 任何一个 nil 都会 panic。
2. 目标是让装配错误在启动阶段暴露，不进入运行期。

### 4.4 rankingInvalidator 抽象

router 会尝试把 rankingRepository 断言为可失效接口：

- InvalidateRankingCache(ctx)

如果断言成功，传给 stall/comment service。

好处：

1. service 层不依赖具体缓存实现。
2. 只依赖一个小接口，降低耦合。

### 4.5 service 与 handler 装配

每个模块都是：

1. NewXxxService(repo...)
2. NewXxxHandler(service)

这样 handler 永远只依赖 service，不直接碰 repository。

### 4.6 v1 路由注册

核心路由：

1. /auth/register login refresh logout
2. /canteens
3. /stalls
4. /stalls/:stallId
5. /stalls/:stallId/ratings
6. /stalls/:stallId/comments
7. /comments/:rootCommentId/replies
8. /comments/:commentId/like (POST/DELETE)
9. /rankings
10. /me/comments
11. /me/ratings

同时你要注意每条路由挂了什么中间件：

1. 需要登录的路由挂 Auth。
2. 游客可访问但需要识别登录态的路由挂 OptionalAuth。
3. 写接口普遍挂 RateLimitByClient。

## 5. 本章你应该能独立写出来什么

读完后你应该能自己实现：

1. 一个带配置加载、日志初始化、依赖装配和 fail-fast 策略的 main。
2. 一个最小可用 migration 执行器。
3. 一个强约束的 router 构造函数（显式注入全部仓储）。
4. 按中间件差异注册受保护和可选鉴权路由。

如果你能复刻这 4 点，后续模块就只是“往骨架里填业务”。
