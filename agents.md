# AGENTS.md — 食堂评分系统开发协作规范

## 1. 目的

本文件定义人类开发者与AI Agent在本项目中的协作边界、执行流程、质量门禁，确保在前后端并行开发时保持一致性、可追踪性与可验证性。

---

## 2. 项目上下文

- 前端：TypeScript + React
- 后端：Go + Gin
- 数据库：MySQL
- 缓存：Redis
- 核心特性：游标分页、楼中楼评论、评论点赞、排行榜筛选

规范基线文件：

1. `frontend-dev.md`
2. `backend-dev.md`
3. `api-spec.md`

若实现与文档冲突，以 `api-spec.md` 为接口准绳，随后同步更新其余文档。

---

## 3. Agent 角色定义

## 3.1 Architect Agent

职责：

- 拆解需求与里程碑
- 制定模块边界与接口优先级
- 识别高风险改动

输出：

- 任务分解表
- 设计决策记录（ADR）

## 3.2 Frontend Agent

职责：

- 实现页面、组件、交互
- 接入游标分页与评论楼中楼
- 落地点赞乐观更新与回滚

必守：

- 不得使用 offset 分页
- 必须复用统一 API SDK

## 3.3 Backend Agent

职责：

- 实现 REST API 与业务逻辑
- 维护评论树关系与点赞幂等
- 保障缓存与DB一致性

必守：

- 列表接口统一 cursor pagination
- 写接口需明确事务边界

## 3.4 QA Agent

职责：

- 编写自动化测试与联调清单
- 验证评论树/点赞/分页正确性

必守：

- 不允许删测试掩盖问题
- 问题必须复现、定位、回归验证

---

## 4. 开发流程（强制）

1. 读取需求与相关文档
2. 输出执行计划（涉及文件、接口、风险）
3. 按模块开发（一个模块一个闭环，变更最小化）
4. 模块完成后立即执行对应测试（至少覆盖该模块相关单元/集成/E2E）
5. 测试通过后再进入下一模块；若模块已达到可交付状态，直接推送到 GitHub
6. 本地验证（lint/test/build）
7. 更新文档（如字段、接口、规则变化）
8. 输出变更说明（影响范围 + 验证结果）

### 4.0 开发文档同步规则（强制，新增）

- 每次完成开发（尤其是每完成一个模块）后，必须同步更新开发文档，至少包含：
  - 已完成功能清单
  - 下一步计划（Next Steps）
  - 待优化事项（Optimization Backlog）
  - 当前阻塞与风险（如有）
  - 继续开发建议（便于下一次会话无缝衔接）
- 必须将以下句子原文写入项目规范并长期保留：
  - **你每次开发完，都要同步更新开发文档，写上你已经完成的功能，下一步要做的，待优化的等等内容，为之后切换对话仍然可以无缝衔接开发做准备。**
- 若本次开发未更新对应模块文档，视为该模块未完成，不允许进入“测试通过后推送”阶段。

### 4.1 模块化交付规则（新增）

- 必须采用“模块开发 -> 模块测试 -> 测试通过 -> 推送 GitHub”流水线
- 任一模块测试失败，禁止推送，先修复后重测
- 模块边界可按业务域（auth/stall/comment/ranking）或前端 feature 目录划分

### 4.2 大模型组件与扩展规则（新增）

- 如大模型开发依赖额外组件、插件、扩展或 SDK，需由执行者自行检索官方文档与可信来源并完成下载/安装
- 新增组件需记录来源、版本与用途，并在相关文档中补充
- 禁止使用来源不明或无法验证安全性的组件

### 4.3 GitHub 提交信息规范（新增）

- 提交前缀仅允许：`feat`、`chore`、`fix`、`refactor`、`docs`
- 提交信息统一使用中文描述，格式：`<type>(可选scope): <中文描述>`
- 示例：
  - `feat: 添加排行榜筛选新特性`
  - `fix: 修复评论点赞计数错误`
  - `refactor: 重构评论服务分层`
  - `chore: 更新开发脚本与依赖`
  - `docs: 更新README`

## 4.4 模型分工规则（新增）

- 前端开发（仅前端）默认使用 Gemini 3.1 Pro Preview（标识：`gemini-3.1-pro-preview`）。
- 除前端开发外的所有开发任务（包括后端、测试、文档、脚本、架构、联调、运维相关），无论使用什么 Agent、什么模式，统一使用 GPT-5.3 Codex xhigh（模型：`gpt-5.3-codex`，推理强度：`xhigh`）。
- 该模型分工规则为全局强约束，需在新建对话中继续沿用，不得因 agent/mode 变化而改变。

## 5. 统一实现约束

## 5.1 游标分页约束

- 所有列表接口必须支持 `limit + cursor`
- 排序键必须稳定（`created_at + id`）
- 响应必须返回 `items/nextCursor/hasMore`

## 5.2 楼中楼评论约束

- 一级评论：`rootId=0,parentId=0`
- 回复评论：`rootId` 指向一级评论
- 查询分两层：一级列表 + 指定一级的回复列表

## 5.3 点赞约束

- 点赞/取消点赞必须幂等
- like_count 与明细表一致（事务或补偿机制）
- 前端必须支持乐观更新失败回滚

## 6. 代码质量门禁

## 6.1 前端门禁

```bash
pnpm lint
pnpm test
pnpm build
```

## 6.2 后端门禁

```bash
go test ./...
go vet ./...
go build ./...
```

## 6.3 联调门禁

- 评论发布、回复、点赞链路全通
- 游标翻页无重复/漏数
- 排行榜筛选结果正确

---

## 7. 禁止事项

1. 禁止绕过 API 文档私自新增字段不登记
2. 禁止引入 `any` 或关闭关键类型检查（前端）
3. 禁止忽略后端错误返回直接吞异常
4. 禁止在未验证的情况下宣称“功能可用”

---

## 8. 文档维护规则

- 接口变更：必须先改 `api-spec.md`
- 交互变更：同步更新 `frontend-dev.md`
- 领域规则变更：同步更新 `backend-dev.md`
- 协作流程变更：更新本文件 `agents.md`

---

## 9. 开发优先级建议

P0：

1. 登录鉴权
2. 窗口列表/详情
3. 评分
4. 楼中楼评论
5. 点赞
6. 排行榜

P1：

1. 监控与指标
2. 敏感词过滤
3. 热门排序优化

---

## 10. 交付定义（DoD）

- 业务：评分、楼中楼、点赞、排行榜、游标分页全量完成
- 质量：前后端门禁命令全部通过
- 文档：三份开发文档与本规范一致无冲突
- 演示：可完成“登录→打分→评论→回复→点赞→排行榜筛选”全链路
